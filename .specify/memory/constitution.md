# Obsidian MCP Notes Plugin Constitution

## Core Principles

### I. Obsidian Plugin Boundaries
All features MUST run as an Obsidian community plugin written in TypeScript and
bundled to a top-level `main.js`. `src/main.ts` MUST stay focused on plugin
lifecycle, settings loading, command registration, and cleanup. Feature logic
MUST live in focused modules under `src/`, with clear boundaries for MCP tools,
vault access, settings, commands, UI, and utilities. Runtime dependencies MUST
be browser-compatible unless the manifest explicitly marks the plugin desktop-only.

Rationale: Obsidian plugins are loaded inside the user application, so startup,
unload behavior, and mobile compatibility depend on small lifecycle code and
well-contained modules.

### II. MCP Tool Contract Clarity
Every MCP-facing capability MUST have a documented tool contract before
implementation. Tool contracts MUST define the tool name, purpose, input schema,
output shape, error cases, and vault permissions needed. The initial product
scope is limited to searching notes, retrieving note documents, and appending
explicit change notes to existing notes. New MCP tools MUST be added only when
they can be tested independently and do not overlap an existing tool.

Rationale: MCP tools become integration surface area for external agents. Stable
contracts prevent accidental behavior changes and make validation concrete.

### III. Vault Privacy and User Consent
The plugin MUST default to local vault operation. It MUST NOT send note content,
filenames, metadata, or user identifiers to any external service unless the user
enables that behavior in settings and the README discloses the exact data flow.
The plugin MUST NOT execute remote code, fetch scripts for evaluation, or update
itself outside normal Obsidian plugin releases. Write operations MUST be limited
to user-requested note changes inside the vault and MUST preserve existing note
content unless the tool contract explicitly allows a targeted modification.

Rationale: Obsidian vaults commonly contain private knowledge. Privacy and write
safety are product requirements, not optional hardening.

### IV. Incremental, Testable Delivery
Work MUST be planned as independently demonstrable user stories. The MVP MUST
deliver one complete MCP workflow before lower-priority features are started.
For each implemented story, validation MUST include lint/build checks and a
manual Obsidian plugin check when the change affects plugin loading, commands,
settings, or vault write behavior. Error handling for missing files, ambiguous
matches, permission denial, and malformed MCP input MUST be included with the
story that introduces the behavior.

Rationale: A plugin that loads but cannot safely complete a single workflow is
not useful. Smaller verified slices reduce breakage in the user vault.

### V. Release Artifact Discipline
Release readiness MUST verify that `manifest.json`, `versions.json`, and
`package.json` versions remain aligned, that `manifest.json` uses a stable plugin
ID, and that `main.js` is generated by `npm run build`. Source control MUST NOT
commit `node_modules/` or other dependency caches. If release artifacts are
committed for this plugin, they MUST be regenerated from the current source and
attached individually to any GitHub release.

Rationale: Obsidian discovers plugins through a small set of root-level files.
Version drift or stale bundles makes local testing and community releases
unreliable.

## Technical Constraints

The project uses npm, esbuild, TypeScript, and the `obsidian` type definitions.
Plans MUST prefer existing Obsidian APIs for vault reads, writes, commands,
settings, notices, modals, and event registration. Long-running or broad vault
scans MUST be deferred until the user invokes a command or MCP tool, and they
MUST be debounced, scoped, or cached when repeated. All listeners, intervals,
DOM handlers, and workspace events MUST be registered through Obsidian cleanup
helpers so plugin unload does not leak behavior.

## Development Workflow

Feature specs MUST state the user-facing workflow, MCP tool behavior, privacy
impact, write safety rules, and measurable acceptance criteria. Plans MUST pass
the constitution check before research or design. Tasks MUST use exact file
paths, keep `src/main.ts` minimal, include settings and README updates when
behavior is configurable or privacy-sensitive, and include validation tasks for
`npm run lint`, `npm run build`, and relevant manual Obsidian checks.

## Governance

This constitution supersedes generic sample-plugin guidance when planning or
implementing features in this repository. Amendments require an update to this
file, a semantic version bump, and a sync check against Spec Kit templates and
runtime guidance docs. Versioning follows semantic rules: MAJOR for removing or
redefining principles, MINOR for adding or materially expanding principles, and
PATCH for wording-only clarifications. Every generated plan MUST record how the
feature satisfies or intentionally violates each principle; violations require a
specific justification and a simpler rejected alternative.
